The Navigation Challenge

A robot navigating a room needs to answer three questions on every iteration of its loop:

👀

Sense

What does the environment look like right now?

🧠

Decide

Given what the sensor says, what should I do?

⚙️

Act

Drive forward, turn, or reverse — then repeat.

Strategy 1 — Reactive (today)

Drive forward. When the forward sensor drops below a threshold, do a quick left/right check and turn toward the more open side. Fast and simple.

Strategy 2 — Full Sweep (Lesson 14)

Stop and sweep the full 180°, storing all readings in an array. Find the nearest obstacle and clearest path from the data. More thorough, slower.

Strategy 1: Reactive / Threshold

The Idea

The servo faces forward (90°) constantly. The sensor reads the distance ahead on every loop. As long as it's clear, the robot drives forward.

When the reading drops below THRESHOLD (e.g., 30 cm), the robot stops, sweeps left then right to compare distances, then turns toward the more open side.

Pros

  • Fast — no waiting for a full 180° sweep
  • Constant forward motion keeps momentum
  • Easy to tune with just four constants
  • Works well in open corridors

Cons

  • Only "sees" forward until an obstacle is close
  • Can miss obstacles approaching from the side
  • Turn amount is time-based — surface friction matters
  • Equal left/right distances cause a fixed-direction fallback

Best When

  • Obstacles are stationary and roughly forward-facing
  • Open room with clear paths on left or right
  • Speed of navigation is a priority

The Decision Loop

┌─────────────────────┐ │ Read forward dist │ └──────────┬──────────┘ │ dist > THRESHOLD? ┌──────┴──────┐ YES NO │ │ Drive Stop Forward Motors │ │ │ Check Left │ Check Right │ │ │ ┌─────────┴──────────┐ │ Left>Right? Right>Left? Equal? │ │ │ │ │ Turn Turn Reverse │ Left Right + Turn Left │ └─────────┬──────────┘ └─────────────┘ (repeat)

Timing in the Loop

  • Clear path: read sensor → drive 200 ms → repeat (~5 checks/sec)
  • Obstacle: stop → look left (500 ms) → look right (500 ms) → return center (300 ms) → turn → continue

The Deadlock Case

If left and right distances are equal, the robot can't choose a side. It reverses briefly then turns left — breaking the symmetry and escaping the corner.

Important

The robot is blind to side obstacles while driving forward. If a wall is closing in from the side, it won't react until that wall is directly ahead. This is what the full-sweep approach (Lesson 14) solves.

Four Constants to Tune

All robot-specific adjustments happen here — you should not need to change anything else in the code.

Constant Default What it controls Tune when…
THRESHOLD 30 cm How close an obstacle must be before the robot reacts Robot reacts too early or runs into things before stopping
DRIVE_SPEED 150 Motor power (0–255) while driving forward Robot is too fast/slow to navigate safely
TURN_SPEED 180 Motor power (0–255) during turns Turns are sluggish or wheels slip badly
TURN_TIME 400 ms How long to run motors during a turn (~90° target) Robot under- or over-rotates on turns

Starting Point

Start with DRIVE_SPEED = 120 and THRESHOLD = 35 for your first run. Give yourself room to make mistakes before optimizing for speed.

Surface Matters

TURN_TIME depends entirely on your surface. Tile ≠ carpet ≠ foam. Always re-tune when you change floors. A good target is a clean 90° turn.

Checking sides..."); scanServo.write(180); delay(500); int leftDist = readDistance(); Serial.print("Left: "); Serial.print(leftDist); Serial.println("cm"); scanServo.write(0); delay(500); int rightDist = readDistance(); Serial.print("Right: "); Serial.print(rightDist); Serial.println("cm"); scanServo.write(90); delay(300); if (leftDist > rightDist) { Serial.println("Decision: TURN LEFT"); turnLeft(TURN_SPEED); delay(TURN_TIME); stopMotors(); } else if (rightDist > leftDist) { Serial.println("Decision: TURN RIGHT"); turnRight(TURN_SPEED); delay(TURN_TIME); stopMotors(); } else { Serial.println("Decision: REVERSE"); reverse(DRIVE_SPEED); delay(600); stopMotors(); turnLeft(TURN_SPEED); delay(TURN_TIME); stopMotors(); } } Serial.println("-------------------"); } void driveForward(int speed) { digitalWrite(LEFT_MOTOR_INA, HIGH); digitalWrite(LEFT_MOTOR_INB, LOW); analogWrite(LEFT_MOTOR_PWM, speed); digitalWrite(RIGHT_MOTOR_INA, HIGH); digitalWrite(RIGHT_MOTOR_INB, LOW); analogWrite(RIGHT_MOTOR_PWM, speed); } void reverse(int speed) { digitalWrite(LEFT_MOTOR_INA, LOW); digitalWrite(LEFT_MOTOR_INB, HIGH); analogWrite(LEFT_MOTOR_PWM, speed); digitalWrite(RIGHT_MOTOR_INA, LOW); digitalWrite(RIGHT_MOTOR_INB, HIGH);analogWrite(RIGHT_MOTOR_PWM, speed); } void turnLeft(int speed) { digitalWrite(LEFT_MOTOR_INA, LOW); digitalWrite(LEFT_MOTOR_INB, HIGH); analogWrite(LEFT_MOTOR_PWM, speed); digitalWrite(RIGHT_MOTOR_INA, HIGH); digitalWrite(RIGHT_MOTOR_INB, LOW); analogWrite(RIGHT_MOTOR_PWM, speed); } void turnRight(int speed) { digitalWrite(LEFT_MOTOR_INA, HIGH); digitalWrite(LEFT_MOTOR_INB, LOW); analogWrite(LEFT_MOTOR_PWM, speed); digitalWrite(RIGHT_MOTOR_INA, LOW); digitalWrite(RIGHT_MOTOR_INB, HIGH);analogWrite(RIGHT_MOTOR_PWM, speed); } void stopMotors() { digitalWrite(LEFT_MOTOR_INA, LOW); digitalWrite(LEFT_MOTOR_INB, LOW); analogWrite(LEFT_MOTOR_PWM, 0); digitalWrite(RIGHT_MOTOR_INA, LOW); digitalWrite(RIGHT_MOTOR_INB, LOW); analogWrite(RIGHT_MOTOR_PWM, 0); } long readDistance() { digitalWrite(trigPin, LOW); delayMicroseconds(2); digitalWrite(trigPin, HIGH); delayMicroseconds(10); digitalWrite(trigPin, LOW); long duration = pulseIn(echoPin, HIGH, 30000); if (duration == 0) return 200; return duration * 0.034 / 2; Tuning Your Robot

Tune in this order — each step depends on the one before it.

Step 1 — Verify the sensor

Upload the code with motors unplugged. Open Serial Monitor at 115200 baud. Hold your hand in front at different distances and confirm the readings make sense before anything moves.

Step 2 — Set THRESHOLD

Watch "Forward: Xcm" in the serial output. Place an obstacle where you want the robot to react and note the distance reading. Set THRESHOLD to that value plus a few cm of margin.

Step 3 — Set DRIVE_SPEED

Plug in motors. Start at 120 and increase until the robot moves confidently without being too fast to react. Most robots land between 130–170.

Step 4 — Tune TURN_TIME

Place the robot and trigger a turn (put an obstacle in front). Measure how many degrees it rotates. Adjust TURN_TIME proportionally:

If it turns 45° with 400 ms → target is 800 ms for 90°.
If it turns 135° with 400 ms → target is ~270 ms.

Step 5 — Set TURN_SPEED

If the motors stall (buzz but don't move) during turns, increase TURN_SPEED. If they skid and overshoot, lower it. A higher speed means you may need a shorter TURN_TIME.

Re-tune when you change surfaces

Friction varies dramatically between tile, carpet, and foam. TURN_TIME in particular will need to be re-established on a new surface.

Troubleshooting

Symptom Likely Cause Fix
Robot drives backward instead of forward Both motor directions wired in reverse Swap INA ↔ INB for both motors in driveForward(), or physically swap the motor wires
Robot turns opposite direction Left/right motors wired opposite to what code expects Swap INA ↔ INB for one motor, or swap the turnLeft() and turnRight() function names
One motor spins, one doesn't PWM pin not connected, or motor driver channel wiring issue Check the PWM pin connection and motor driver power supply
Sensor always reads 200 cm Trigger or echo pin wired to wrong pin Verify trigPin and echoPin match your wiring
Robot reacts to obstacles that aren't there THRESHOLD too high, or sensor noise Lower THRESHOLD; ensure sensor is mounted flat and not picking up the chassis
Servo doesn't move Pin 18 doesn't match A4 on your board Try other nearby integers (17, 19) and confirm with a simple scanServo.write(90) test sketch

What to Watch For

  • Forward readings should be stable (±2–3 cm is normal)
  • 200 cm always means a timeout — no echo returned
  • Left and right readings during a check should reflect what's actually on each side of the robot
  • If decisions don't match reality, the servo may not be reaching 0° or 180° — check TURN_TIME vs servo speed

Wall-Following: Tuning the P-Controller

error = dist − TARGET_DIST  |  rightSpeed = BASE_SPEED − (Kp × error)

Tuning Order

  1. Set TARGET_DIST first — hold the sensor at your desired wall distance and confirm that reading in Serial Monitor before the robot moves.
  2. Set BASE_SPEED low (120–130) for first runs. Slower movement gives corrections more time to take effect.
  3. Start Kp at 1.0. The robot will follow loosely. Increase by 0.5 at a time until it tracks the wall tightly.
  4. If it oscillates (weaves back and forth), Kp is too high — reduce it by 0.5.
  5. Once stable, increase BASE_SPEED gradually for a faster run.

Reading the Serial Output

Dist: 18cm Err: 3 R: 131 ← drifting away, correcting right Dist: 15cm Err: 0 R: 140 ← on target, both motors equal Dist: 11cm Err: -4 R: 152 ← too close, steering away
  • Error near 0 — robot tracking correctly
  • Error swinging ±5+ — Kp too high, reduce it
  • R speed hitting 0 or 255 — correction is maxing out; lower Kp or BASE_SPEED
  • Dist always 200 — sensor timeout; wall is out of range or servo angle is wrong

Common Problems

  • Robot curves away from wall immediately — swap which motor is constant vs. adjusting, or flip the sign on Kp
  • Robot hugs wall and grinds — TARGET_DIST too small or Kp too high
  • Works on straight wall, fails at corners — expected; P-only can't anticipate. A corner fix requires adding obstacle detection from Lesson 13
