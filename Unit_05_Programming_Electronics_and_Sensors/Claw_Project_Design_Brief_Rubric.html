Design Brief Rubric

Unit 5 — Claw Project | Deliverables 8 & 9

Document: Design Brief (Google Doc)
Checkpoint (D8): Sections 1–4 + AI Log — 50 pts
Final (D9): All Sections — 25 pts

Overview

The Design Brief is a 9-section technical document students build progressively over the 10-day project. It is submitted twice: a Week 1 Checkpoint (Deliverable 8, Day 5) covering Sections 1–4 and the AI Log, and a Final submission (Deliverable 9, Day 10) covering all 9 sections. Grading emphasizes accuracy and understanding over polish — students should write in their own words, not copy AI output verbatim. Annotated code must show the student's own explanations of every function.

★
Checkpoint grading (Deliverable 8 — 50 pts): Grade only Sections 1–4 (40 pts) + AI Log (10 pts). Sections 5–9 are not yet expected; do not penalize for absence.

Final grading (Deliverable 9 — 25 pts for Design Brief portion): Grade Sections 5–9 (25 pts total). Sections 1–4 are already graded from D8; re-grade them only if significantly revised.
Deliverable 8 — Checkpoint Sections (50 pts)
Section 1 — Project Overview (6 pts)
Criterion Performance Description Pts
Purpose & Context
3 Clearly states what the project is, what the system does (grip, classify, detect slip), and why it is a useful application of sensors and embedded programming. Written in 2–4 sentences in student's own voice.
2 States what the project is but description of functionality is vague or incomplete.
1 Present but reads like AI output pasted without editing; no personal voice.
0 Missing or one-sentence placeholder.
 3
Learning Goals
3 Lists 2–3 specific learning goals (e.g., "I will learn how PWM controls servo position" and "I will learn how to use ADC values to detect contact"). Goals are specific, not generic.
2 Goals listed but generic ("I will learn programming").
1 One goal present; vague.
0 Missing.

Section 2 — Bill of Materials (8 pts)
Criterion Performance Description Pts
Completeness
4 All major components listed (microcontroller, servo, potentiometer, LED(s), resistor(s), claw hardware, power supply). No significant omissions.
2–3 1–2 components missing; core items present.
1 Significant omissions; fewer than half the components listed.
0 Missing or empty.
 4
Datasheet Links
4 Working hyperlinks to datasheets (or product pages) provided for servo, potentiometer, and at least one other component. Links actually open the correct resource.
2–3 Links present for 2 components; one broken or incorrect link.
1 One working link; others missing or broken.
0 No links.

Section 3 — Hardware Setup (8 pts)
Criterion Performance Description Pts
Pin Table Accuracy
5 Pin table lists all connected components with correct pin numbers/names, wire color, and signal type (power/signal/ground). Matches actual wiring (green = pot, white = servo sig, red = power, black = ground).
3–4 Table present with minor errors (1–2 wrong pin numbers or missing a column).
1–2 Table attempted but significant errors or missing components.
0 No pin table.
 5
Diagram or Photo
3 A wiring diagram or clear photo is included that matches the pin table and shows connections legibly.
2 Diagram/photo present but unclear or partially matching.
1 Very blurry photo or a generic diagram not specific to this project.
0 Missing.

Section Total/ 8
Section 4 — Key Concepts (18 pts) ★ Highest-weighted section
Criterion Performance Description Pts
PWM Explanation
6 Explains in student's own words: what PWM stands for, what duty cycle means, how changing duty cycle controls servo position. Uses an analogy or example. Does not read like a textbook or AI dump.
3–5 Explanation is mostly correct but incomplete (e.g., defines PWM but doesn't connect to servo angle) or is clearly paraphrased AI with no personal voice.
1–2 Definition only ("PWM is a technique...") with no connection to the servo or claw.
0 Missing or completely wrong.
 6
ADC Explanation
6 Explains in student's own words: what ADC does (analog → digital), what the 0–1023 range represents, and how ADC values from the potentiometer are used to detect claw position/contact. Example or analogy included.
3–5 Explanation mostly correct; missing connection to the potentiometer's role in claw control, or lacks personal voice.
1–2 Surface definition only ("ADC converts analog signals to digital") with no application to project.
0 Missing or completely wrong.
 6
Additional Concept
6 Student includes and explains at least one additional concept relevant to their implementation (e.g., control loop, threshold logic, serial communication, mapping function, or slip detection algorithm). Explanation is accurate and in their own words.
3–5 Additional concept present but explanation is shallow or partially accurate.
1–2 Names a concept without explaining it.
0 No additional concept beyond PWM and ADC.
 6
Section Total/ 18
Section 8 — AI Conversation Log, Checkpoint (10 pts)
Criterion Performance Description Pts
Prompt Count & Variety
5 At least 3 prompts logged by Day 5. Prompts cover different topics (e.g., not three variations of the same question). Prompts are real (match the work done on the project).
3–4 2 prompts logged, or 3 prompts that are very similar to each other.
1–2 1 prompt logged, or prompts clearly fabricated.
0 No AI log entries.
 5
Reflection on AI Use
5 For each logged prompt, student notes what AI got right, what it missed or got wrong, and what the student did with the output (accepted, modified, ignored). At least one entry notes a case where AI was incorrect or vague.
3–4 Some reflection present but not for every entry, or all reflections are positive with no critical evaluation.
1–2 Log just lists prompts with no reflection.
0 No reflection.

Deliverable 9 — Final Sections (25 pts)
Section 5 — Algorithm (6 pts)
Criterion Performance Description Pts
Flowchart or Pseudocode
4 Flowchart or pseudocode accurately represents the main program loop: open claw → read ADC → check contact → classify size → set LED → check for slip → repeat. Decision branches are present. Matches actual code.
2–3 High-level steps present but missing decision branches or does not match code.
1 Very rough sketch; missing key steps.
0 Missing.
 4
Clarity
2 Another student could follow the algorithm to understand the program flow without reading the code.
1 Understandable with effort; some ambiguity.
0 Unclear or inaccurate.
 2
Section Total/ 6
Section 6 — Annotated Code (8 pts)
Criterion Performance Description Pts
Annotation Quality
5 Every function or major block has a comment written by the student explaining what it does and why — not just a restatement of the code. Comments are in student's own words (not AI-generated boilerplate).
3–4 Most functions annotated; some comments just restate the code ("this reads the ADC value" with no explanation of why or what happens next).
1–2 Sparse annotations; fewer than half the functions commented.
0 No annotations, or code is not included.
 5
Code Accuracy
3 Code in the Design Brief matches (or is a clean final version of) the code that ran during the live demo. No placeholder functions left unexplained.
2 Minor discrepancies between Design Brief code and demo code.
1 Code is incomplete or significantly different from what was demoed.
0 No code or clearly not the student's work.

Section Total/ 8
Section 7 — Testing Results (5 pts)
Criterion Performance Description Pts
Data Table
3 Table includes results for ≥ 5 objects. Columns: object name, estimated size, ADC reading at contact, classified size, correct? Rows are filled in with real values (not invented round numbers).
2 Table present with 3–4 objects, or some columns missing.
1 Fewer than 3 objects, or data looks fabricated.
0 No testing results.
 3
Analysis
2 Student notes any objects that were misclassified and explains why (e.g., "the ping pong ball was light enough that the ADC threshold I used put it in the wrong category"). Proposes one improvement.
1 Analysis present but superficial ("it worked most of the time").
0 No analysis.

Section Total/ 5
Section 9 — Challenges & Solutions (6 pts)
Criterion Performance Description Pts
Challenges Identified
3 At least 2 real challenges described with specifics (e.g., "The ADC readings fluctuated ±30 when the claw was stationary, causing false contact triggers"). Challenges are plausible and match the complexity of the project.
2 One specific challenge plus one vague challenge ("programming was hard").
1 Vague or generic challenges only.
0 Missing.
 3
Solutions Documented
3 For each challenge, student explains the solution they tried and whether it worked. Unresolved challenges are noted honestly. Shows a real debugging/iteration process.
2 Solutions listed but not connected clearly to specific challenges, or "I googled it" without detail.
1 Solutions very vague or missing for some challenges.
0 No solutions documented.
